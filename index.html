The Problem

Digital repression is physical

In countries like Belarus, Russia, Iran, Myanmar, China, and others — as well as in occupied territories of Ukraine — authorities don't just monitor online traffic. They confiscate devices on the street, during home raids, and at border crossings, then use forensic tools to extract browsing history, app data, and subscriptions. People are sentenced to prison for old chat messages. Even a subscription to an independent news channel can trigger prosecution. End-to-end encryption solves network interception — but it does not protect users when their unlocked device is in someone else's hands.

📱

Device inspections

At checkpoints, borders, and during street stops, police demand to unlock devices on the spot. Any "suspicious" app, subscription, or chat history becomes evidence. Even if a secure app is deleted, the operating system retains logs of its name and usage time — permanent evidence that the app was used.

🔬

Forensic extraction

Authorities use specialized tools (Cellebrite, Oxygen Forensic) to bypass OS protections and recover deleted data, browser caches, and metadata. E2EE protects transit — but local encrypted databases become readable once the device is unlocked or the OS bypassed.

The Solution

Nothing to install. Nothing to find.

in6.to is a browser-based platform that runs entirely in incognito mode over a VPN connection, enforcing both as hard requirements. There is no app, no extension, no file on the device. When the user closes the tab, the session is destroyed and no recoverable data remains.

The core insight

Secure messengers try to hide data on the device — behind encryption, behind secret chats, behind disappearing messages. But the device still knows the app exists. Operating system logs record its name and usage time. Forensic tools find its database automatically. The app icon itself invites suspicion.

in6.to takes a different approach: remove the data from the device entirely. The device becomes a screen, not a vault. The operating system sees only a normal browser. There is no app to find, no database to extract, no usage pattern that stands out from millions of ordinary browsing sessions.

The platform is built for open-source release. The codebase will allow organizations to deploy independent instances if needed.

How in6.to relates to existing tools

in6.to does not replace other security tools — it fills a specific gap: what happens when someone physically takes your device.

🔒

E2EE messengers (Signal, Telegram, Briar, Session)

Protect message content in transit. But the app is visible on the device, its database is recoverable with forensic tools, and the operating system permanently logs its name and usage time — even after deletion. in6.to lets users access these messengers remotely, through a normal browser that leaves no distinct footprint. The approaches are complementary.

💻

Privacy-focused OSes (Tails, Whonix, Qubes OS)

Excellent solutions — Tails leaves no traces by design. But they require a separate USB drive or dedicated hardware, technical setup, and cannot run on a phone. in6.to works in any browser on any device, including mobile. For advanced users, the tools can be combined for layered protection.

☁️

Remote browsers (Authentic8 Silo, Kasm Workspaces)

Enterprise tools that isolate browsing from the local device. They solve a similar technical problem but are designed for corporate IT security, not for people under physical threat. They require organization accounts and lack features like instant session destruction on movement, decoy accounts, or identity-free authentication.

Modular architecture

Any web application can be added as a module via nginx reverse proxy and iframe isolation. The platform intercepts and encrypts each app's browser storage (cookies, localStorage, sessions), saving them server-side. On next login, these states are decrypted and injected back — so the user logs into a web app once, and it stays authenticated across sessions without ever touching the device's real storage.

Modules can be private and invisible to other users. Each user or organization can have their own set enabled.

Security Architecture

Defense in depth, designed for the worst case

Every layer is built around a single question: what happens if they take the device right now? The platform is at MVP stage — core protections are implemented, with further hardening underway.

Technical stack

Frontend: single-page React application with in-memory state only — no persistent storage APIs used. Backend: Ruby on Rails API server behind Cloudflare. External apps (Telegram Web, Guacamole, Parsec) are loaded in sandboxed iframes with per-session tokens. Authentication keys are derived client-side; the server stores only encrypted blobs it cannot decrypt. All communication over TLS. Infrastructure designed for UI/API separation — static frontend served independently (e.g. via Cloudflare Pages) to prevent UI tampering even if the backend is compromised.

✓ Implemented

01

Client-side encryption for Telegram sessions

Encryption keys are derived on the device from user credentials. Telegram session data is encrypted locally before being sent to the server. By design, the API server receives and stores only encrypted blobs it cannot decrypt.

02

No phone, no email, no identity

Authentication uses a numeric code and a graphical pattern — no personal identifiers required. Any combination creates a separate account. Currently ~10 billion possible combinations; planned upgrade to a quadrillion.

03

Aggressive session termination

Sessions are destroyed automatically upon: inactivity (60s), tab losing focus, sudden device movement (phone grab scenario), and failure to re-authenticate during periodic checks.

04

Zero browser storage

No localStorage, sessionStorage, cookies, or service workers. All state lives in JavaScript memory only. Aggressive cache-busting headers (Cache-Control: no-store, Clear-Site-Data) on all responses.

05

Token security

Session tokens are bound to IP address and device fingerprint. Any change triggers immediate invalidation. Tokens are deactivated server-side upon logout, inactivity, or connection loss.

06

Iframe isolation

Login occurs in an isolated iframe that is destroyed after authentication, releasing credentials from memory. External apps run in sandboxed iframes with controlled, temporary tokens.

07

Decoy accounts

Incorrect re-authentication locks the original account and presents an empty decoy. This protects against coerced login — entering a wrong pattern deliberately shows an empty account.

08

Enforced VPN + incognito

The platform refuses to operate without both. This is a hard requirement, not a recommendation — ensuring baseline protection for every user.

09

Full browser history erasure

All navigation state is kept in an in-memory router. On logout, location.replace() redirects to an innocent site (e.g. google.com). The browser's back button shows no trace of the platform — only a normal site visit.

10

Server-side connection keepalive

A persistent connection to the server ensures that if the device goes offline — whether from network loss or physical seizure — the token is automatically deactivated server-side within seconds.

11

Cloudflare protection

All traffic is routed through Cloudflare to prevent exposure of server IP addresses and provide DDoS protection. Server access is restricted to Cloudflare-originating requests only.

12

Anti-fake monitoring

A network of automated bots monitors hundreds of Telegram chats for mentions and phishing attempts, instantly warning users about fake copies of the platform or social engineering attacks.

Threat model boundaries

The platform is designed to protect against physical device inspections, forensic extraction of local data, and coerced access. It does not claim to protect against all possible threats. We are transparent about the boundaries:

OS-level compromise. If the operating system itself is compromised — through pre-installed spyware, a compromised firmware, or a hardware keylogger — no browser-based solution can fully protect the user. The platform's protection boundary ends at the OS level.

Memory forensics. While JavaScript memory is cleared on logout, operating system swap files or hibernation images could theoretically retain fragments. We provide guidance to advanced users on disabling these, but it is not enforceable from the browser.

Centralized infrastructure. The platform currently depends on a central server. This is a known single point of failure. The roadmap includes self-hosted deployments for NGOs and mesh distribution to mitigate this.

Not yet audited. The platform has not undergone a professional third-party security audit. This is a planned deliverable — see the roadmap.

How It Works

From opening the browser to full protection

Designed to be as simple as possible for non-technical users. No configuration, no installation, no technical knowledge required.

1

Open a browser in incognito mode with VPN enabled

Even a built-in VPN (like Opera's) is sufficient. The platform verifies both conditions before proceeding.

2

Authenticate with a visual pattern

Select 6 objects from a detailed illustration with hundreds of items. This sequence is your identity and encryption key source. No phone number, no email — nothing traceable.

3

Access your protected workspace

Your encrypted session is decrypted client-side. Use Telegram, remote desktops, Android emulator, or the internal messenger — all within the same interface.

4

Work, communicate, read the news

Everything stays within the platform. Session states are encrypted and stored server-side so you can resume next time. The server sees only encrypted data.

5

Close the tab — everything vanishes

The session is destroyed, browser history is replaced with an innocent site, the server-side token is invalidated, and memory is cleared. Nothing remains on the device.

FAQ

Common questions

Why not just use an encrypted messenger?

E2EE messengers protect data in transit but not against device seizure — all chats, contacts, and history are visible once the phone is unlocked. The app icon itself raises suspicion. in6.to ensures there is nothing on the device to find. The two approaches are complementary — you can use encrypted messengers inside the platform.

Can I use this at a border crossing?

This is a key scenario. You keep ordinary apps on your phone while sensitive accounts exist only inside the platform. At a border crossing, your device looks completely normal — no suspicious apps, no browsing history, nothing to prompt additional questions.

Is this a complete guarantee of safety?

No. The platform cannot defend against OS-level exploits, hardware keyloggers, or compromised firmware. What it provides is a high minimum floor of protection for non-technical users: enforced VPN and incognito mode, automatic session destruction, zero traces in the browser, and defense against common physical inspection scenarios. See the Known Limitations section for a detailed breakdown.

What happens if someone forces me to show my browser?

Incognito mode means there is nothing to show. Sessions terminate after 60 seconds of inactivity, on tab blur, and on sudden movement. The browser redirects to google.com — no back-button trail. If forced to open the platform, any code/pattern combination opens a new empty account.

Why require VPN and incognito?

VPN prevents the ISP from seeing that you accessed the platform. Incognito ensures no cache, cookies, or history after the tab is closed. By enforcing both as hard requirements, we guarantee a baseline for every user. A future version will use traffic masking to eliminate the VPN requirement entirely.

When will the code be open-sourced?

Open-source release is a planned deliverable. We are preparing a stable, well-documented release that the community can meaningfully review and build upon. We are committed to full transparency and want the published code to meet a quality standard worthy of community trust.
