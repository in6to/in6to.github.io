A zero-trace platform for secure communication in authoritarian states

in6.to is a browser-based platform that leaves nothing behind on the device. No app to install. No data to find. Close the tab ‚Äî and everything disappears. Built for people in Belarus, Russia, and other repressive environments where accessing independent information or communicating freely can lead to prosecution. The goal is simple: let people exercise their basic rights without risking their safety.

The Problem

Digital repression is physical

In countries like Belarus, Russia, Turkmenistan, Iran, and others, authorities don't just monitor online activity ‚Äî they confiscate devices on the street, during home raids, and at checkpoints. People are being sentenced to years in prison for old comments in Telegram chats. Even a subscription to an opposition channel can trigger criminal prosecution. People need a way to communicate, read independent news, and organize ‚Äî without leaving any trace on their device.

üö™

Home raids

Security forces break into homes and seize all electronics for forensic analysis. Hard drives, phones, and tablets are examined using professional extraction tools.

üì±

Street inspections

Police stop people on the street or at work and demand to unlock their devices on the spot. Any "suspicious" app, subscription, or chat history becomes evidence.

üî¨

Forensic extraction

Authorities use specialized hardware and software (Cellebrite, Oxygen Forensic, etc.) to bypass OS protections and extract deleted data, browser caches, and metadata.

The Solution

Nothing to install. Nothing to find.

in6.to runs entirely in the browser's incognito mode over a VPN connection. It enforces both ‚Äî refusing to operate without them. There is no app, no extension, no file on the device. When the user closes the browser tab, the session is destroyed and no recoverable data remains on the device.

Security Architecture

Defense in depth, designed for the worst case

Every layer of the platform is built around a single question: what happens if they take the device right now? The answer should always be: nothing useful is found. The platform is at MVP stage ‚Äî core protections are implemented and working, with further hardening underway.

‚úì Implemented

01

Client-side encryption for Telegram sessions

Encryption keys are derived on the device from user credentials. Telegram session data is encrypted locally before being sent to the server. The server stores only encrypted blobs and cannot read session contents.

02

No phone, no email, no identity

Authentication uses a numeric code and a graphical pattern ‚Äî no personal identifiers required. Any combination creates a separate account. Currently ~10 billion possible combinations; planned upgrade to a quadrillion (see roadmap).

03

Aggressive session termination

Sessions are destroyed automatically upon: inactivity (60 seconds), tab losing focus, sudden device movement (phone grab scenario), and failure to re-authenticate during periodic checks.

04

Zero browser storage

No localStorage, sessionStorage, cookies, or service workers. All state lives in JavaScript memory only. Aggressive cache-busting and storage-clearing HTTP headers (Cache-Control: no-store, Clear-Site-Data) on all responses.

05

Token security

Session tokens are bound to the current IP address and device fingerprint. Any change triggers immediate invalidation. Tokens are deactivated server-side upon logout, inactivity, or connection loss.

06

Iframe isolation

Login occurs in an isolated iframe that is destroyed after authentication, releasing credentials from memory. External apps run in sandboxed iframes with controlled, temporary tokens.

07

Decoy accounts

If an incorrect re-authentication attempt is made, the original account is locked with exponential backoff, and the user is presented with an empty decoy account instead.

08

Enforced VPN + incognito

The platform refuses to operate without both a VPN connection and incognito/private browsing mode. This is a hard requirement, not a recommendation ‚Äî ensuring baseline protection for every user.

09

Page reload on logout

Every logout triggers a full page reload to force browser garbage collection and clear any remaining JavaScript variables, closures, and DOM references from memory.

10

Cloudflare protection

All traffic is routed through Cloudflare to prevent exposure of server IP addresses and provide DDoS protection. Server access is restricted to Cloudflare-originating requests only.

11

Anti-fake monitoring

A network of automated bots monitors hundreds of Telegram chats for mentions and phishing attempts, instantly warning users about fake copies of the platform or social engineering attacks.

12

Safe-page redirect on logout

On logout, the browser is redirected to an innocent website (e.g. google.com) using location.replace(). Browser history shows only a normal site visit ‚Äî there is no back-button trail leading to the platform.

‚ü≥ In progress / Planned hardening

‚Üí

Full client-side encryption for all modules

Extend client-side encryption beyond Telegram sessions to the internal messenger and all other stored data. Currently, some modules transmit data securely but don't yet encrypt at rest on the server.

‚Üí

Full UI / API server separation

Separate static UI and API onto different servers. Serve the UI as purely static files (e.g. via Cloudflare Pages) to minimize attack surface and prevent UI tampering even if the backend is compromised.

‚Üí

Argon2 key derivation

Replace current SHA-based key derivation with Argon2id for significantly stronger resistance against brute-force attacks on the authentication credentials.

How It Works

From opening the browser to full protection

The entire workflow is designed to be as simple as possible for non-technical users while maintaining maximum security. No configuration, no installation, no technical knowledge required.

1

Open a browser in incognito mode with VPN enabled

Even a built-in VPN (like Opera's) is sufficient. Navigate to the platform URL. The platform verifies both conditions before proceeding.

2

Authenticate with a visual pattern

Select 6 objects from a richly detailed picture (a "wimmelbook" illustration with hundreds of items). This sequence is your identity and encryption key source. No phone number, no email ‚Äî nothing traceable.

3

Access your protected workspace

Your encrypted session is decrypted client-side. Use Telegram, remote desktops, Android emulator, or the internal messenger ‚Äî all within the same protected interface.

4

Work, communicate, read the news

Everything you do stays within the platform. Session states are encrypted and stored server-side so you can pick up where you left off next time. The server sees only encrypted data.

5

Close the tab ‚Äî everything vanishes

On logout (manual, automatic, or forced), the session is destroyed, browser history is replaced, the server-side token is invalidated, and the page reloads to clear memory. Nothing remains on the device.
